# 프로젝트 개요

**HUFS.Dev**는 한국외국어대학교 학생들이 개발 관련 정보와 경험을 공유하고, 함께 성장할 수 있도록 설계한 `대학 개발자 커뮤니티 플랫폼`입니다.

기존에는 개발 스터디, 프로젝트 모집, 기술 질문, 회고와 같은 활동이 여러 채널에 흩어져 있어 정보가 축적되지 않거나 
필요한 사람에게 적절히 연결되지 않는 문제가 있었습니다. 또한 학교 커뮤니티 안에서도 개발자들에게 특화된 소통 공간과 활동 동기를 만들어주는 구조는 부족했습니다.

HUFS.Dev는 다양한 개발 관련 게시판, 랭킹·뱃지, 활동 기반 커뮤니티 시스템을 통해 단순한 정보 게시 공간을 넘어, `기여가 기록되고 활동이 축적되는 개발자 커뮤니티`를 만드는 것을 목표로 했습니다.

# 시스템 아키텍처 설계

![시스템 아키텍처](https://eaalkymxyfskjojh.public.blob.vercel-storage.com/portfolio/hufsdev/hufsdev-architecture.png)

HUFS.Dev의 인프라는 **AWS 서비스를 직접 설계·구성해보며 학습하는 것**을 주요 목표로 두고 설계했습니다.  
단순히 애플리케이션을 배포하는 데 그치지 않고, 실제 서비스 운영에 가까운 형태로 정적 자산 배포, 파일 저장, 리버스 프록시, 블루-그린 배포, 데이터베이스, 캐시 계층을 분리해보며  
각 인프라 컴포넌트의 역할과 연결 구조를 직접 경험하는 데 초점을 맞췄습니다.

### 프론트엔드 배포

프론트엔드는 Vercel에 배포해 정적 자산을 빠르게 제공하도록 구성했습니다. 
프론트 배포를 백엔드 릴리즈와 분리하여 UI 변경을 더 빠르게 반영하고, 
백엔드 배포 리스크에 영향을 받지 않도록 했습니다.

### 백엔드 인프라

백엔드 트래픽 흐름은 **Route 53 → ALB → Nginx → Spring Boot** 구조로 구성했습니다. 
애플리케이션 서버 내부에서는 Nginx가 리버스 프록시로 동작하며, Blue/Green Spring Boot 컨테이너 사이의 
라우팅을 제어하도록 설계했습니다.

데이터 계층은 **PostgreSQL(RDS)** 과 **Redis(ElastiCache)** 로 분리했고, 
파일 업로드는 **S3**, 정적 파일 조회는 **CloudFront** 를 통해 제공하도록 구성했습니다. 
또한 백엔드 이미지는 **GitHub Actions → ECR → EC2 docker pull** 흐름으로 배포되도록 구성했습니다.

### 배포 전략

배포는 **Nginx upstream 전환 기반의 Blue-Green 방식**으로 운영했습니다.  
동일 EC2 내부에서 Blue/Green 애플리케이션 컨테이너를 분리 실행하고, 전환 시점에만 Nginx 라우팅 대상을 변경해 서비스 중단을 최소화하도록 구성했습니다. 
이를 통해 배포 과정에서 새 버전을 안전하게 검증하고, 문제가 발생하면 이전 버전으로 빠르게 복귀할 수 있도록 했습니다.

### + 왜 Blue-Green 배포를 선택했는가...

배포 전략을 설계할 때는 서비스 중단을 줄이고, 문제가 발생했을 때 이전 버전으로 빠르게 되돌릴 수 있는 구조를 우선적으로 고려했습니다.  
당시에는 Blue-Green 배포를 중심으로 먼저 학습하고 적용했는데, 동일한 애플리케이션을 두 개의 버전으로 분리해 두고 라우팅만 전환하면 된다는 점에서 구조가 직관적이고 운영 흐름을 이해하기 쉽다고 판단했습니다.

이후 배포 방식을 더 찾아보면서 Rolling, Canary 등 다양한 전략이 있다는 점도 알게 되었습니다. 
다만 HUFS.Dev의 초기 규모에서는 트래픽을 점진적으로 나누어 검증하는 복잡한 제어보다, `새 버전 준비 → 전환 → 문제 시 복귀`가 명확한 Blue-Green 방식이 더 적합하다고 보았습니다.

### 파일 스토리지 및 CDN

업로드 파일은 S3에 저장하고, 조회 트래픽은 CloudFront를 통해 분산하도록 구성했습니다. 
이를 통해 애플리케이션 서버가 정적 파일 전송까지 직접 담당하지 않도록 분리했고, 
파일 저장소와 애플리케이션 서버의 책임을 나누는 구조를 경험했습니다.

### + 설계 과정에서 얻은 인사이트

이 아키텍처는 학습 목적에 맞게 AWS의 다양한 서비스를 직접 사용해보는 데 초점을 맞춰 구성했습니다. 
다만 실제 운영 과정에서 한 달 인프라 비용이 약 **60달러** 수준까지 발생하면서, 
기술적으로 가능한 구성과 비용 효율적인 구성은 다를 수 있다는 점을 체감했습니다.

특히 **ALB는 당시 학습 목적상 도입했지만**, 현재 돌아보면 이 프로젝트 규모에서는 
Nginx만으로도 충분히 트래픽 진입과 라우팅을 처리할 수 있었고, 
굳이 ALB까지 두지 않아도 되었을 가능성이 크다고 판단했습니다.  

결국, 인프라 설계에서는 기능 구현뿐 아니라 `트래픽 규모`, `운영 복잡도`, `비용 대비 효용`까지 함께 고려해야 한다는 점을 배웠습니다.
